1 · Konzept

Jede HTTP-Route besteht aus drei Bestandteilen, die zusammen eine vollständige Route ergeben:

BestandteilWoBeschreibung
Handler-MethodeShared/DataModuls/*.pasEnthält die eigentliche Logik
Route-KommentarDirekt über der MethodeDokumentiert Route, Auth, LocalOnly
Route-RegistrierungWebModuleUnit1.pas → WebModuleCreateVerbindet URL mit Handler
⚠ Achtung Ohne Eintrag in WebModuleCreate ist eine Methode nicht erreichbar — auch wenn sie vollständig implementiert ist.

2 · Kommentar-Konvention (Pflicht)

Jede öffentliche Handler-Methode muss genau diese Zeile direkt vor der procedure-Deklaration tragen:

// Route: /meinroute  |  Auth: true  |  LocalOnly: true
procedure TDataModulXxx.MeineMethode;
FeldBedeutung
Route: URL-Pfad, wie er in AddRoute registriert wird.
Bei mehreren Pfaden kommagetrennt.
Auth: true JWT-Token erforderlich  |  false öffentlich ohne Login
LocalOnly: true Nur vom Server selbst erreichbar  |  false auch von außen

3 · Neue Route hinzufügen — Schritt für Schritt

Schritt 1 — Handler-Methode implementieren

Wähle das passende DataModul (oder lege ein neues an — siehe Abschnitt 5):

DataModulZweck
DataModulSQLClassGenerische SQL-Operationen (select, insert, update, delete)
DataModulAddOnClassZusatzfunktionen (Geo, KI, CSV-Import, Demos)

Methode im Interface-Abschnitt deklarieren:

  TDataModulXxx = class(TDataModulBaseClass)
  public
    procedure MeineMethode();
  end;

Methode im Implementation-Abschnitt implementieren:

// Route: /meinroute  |  Auth: true  |  LocalOnly: true
procedure TDataModulXxx.MeineMethode;
begin
  Response.ContentType := 'application/json';
  Response.StatusCode  := 200;
  Response.Content     := '{"status":"OK"}';
end;

Schritt 2 — Route registrieren

In Shared/WebModuleUnit1.pas, Methode WebModuleCreate:

FRouter.AddRoute('/meinroute', CreateDataModulXxx,
  TDataModulXxx(nil).MeineMethode,
  {AuthRequired=} true,
  {LocalOnly=}    true);
ParameterTypDefaultBedeutung
Routestring—URL-Pfad, z. B. /meinroute. Wildcard: /prefix/* fängt alle Unterpfade ab.
FactoryTInstanceFactory—Die globale CreateXxx-Funktion der Unit (erzeugt das DataModul)
MethodeTRouteHandler—Methode des DataModuls — wird über Code-Pointer gebunden
AuthRequiredbooleantrueJWT-Prüfung vor Aufruf
LocalOnlybooleanfalseNur von 127.0.0.1 / ::1 erreichbar

4 · Zugriffsstrategie: LocalOnly vs. Public API

TypLocalOnlyWer hat Zugriff?Geeignet für
Intern Standard true Nur Server (127.0.0.1, ::1) Admin, Datenbankzugriffe, Druck, Entwicklungstools
Public API false Angular-Frontend, externe Clients Sichere, whitelistbasierte Endpunkte in DataModulPublicApiClass
Faustregel Neuer Endpunkt für das Angular-Frontend? → In DataModulPublicApiClass, mit LocalOnly: false.
Alles andere? → LocalOnly: true.
Sicherheitshinweis DataModulPublicApiClass: Jedes Feld wird gegen eine ALLOWED-Konstante validiert — kein unkontrollierter SQL-Zugriff.

6 · Neuen DataModul-Controller anlegen

Warum das nicht trivial ist Jeder Controller besteht aus zwei Dateien (.pas + .dfm) und muss an vier Stellen registriert werden — drei davon liegen außerhalb der neuen Datei. Wer eine Stelle vergisst, bekommt einen Laufzeitfehler oder eine nicht erreichbare Route.

Schritt 1 — In Delphi IDE: Neues DataModule anlegen

File → New → Other → DataModule

Delphi erzeugt DataModulXxxClass.pas und DataModulXxxClass.dfm. Danach Schritte 2–6 manuell durchführen.

7 · Checkliste — Vollständig?

  • 1Basisklasse TDataModulBaseClass verwendet (nicht TDataModule) — in .pas

  • 3Factory-Funktion CreateXxx implementiert — in .pas
  • 4Route-Kommentar über jedem Handler — in .pas
  • 5.dfm beginnt mit inherited (nicht object) — in .dfm
  • 6Unit in IBApiTest.dpr eingetragen — in .dpr
  • 7Unit in mod_webbroker.dpr eingetragen — in .dpr
  • 8Unit in uses von WebModuleUnit1.pas — in .pas
  • 9Routen in WebModuleCreate registriert — in WebModuleUnit1.pas
Alle 9 Punkte erledigt? Route ist aktiv. Starte IBApiTest.exe (Standalone) oder deploye die DLL — der neue Endpunkt ist sofort erreichbar.
